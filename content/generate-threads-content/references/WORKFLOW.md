# Threads Content Workflows

Always read VOICE.md before running either workflow. The voice profile and review rubric defined there govern every step below.

---

## Workflow 1 — Discovery and Brainstorming

Use this when the user wants to find something to post about. They may have a loose signal (a mood, a topic, something they saw) or nothing at all.

### Step 1 — Capture the spark
Ask the user for a signal. This can be:
- A topic or theme ("something about design systems")
- A feeling or moment ("I just sat through a terrible design review")
- A recent experience ("I drove the Rivian this weekend")
- Nothing ("just help me find something to post")

If they give you nothing, suggest 3 topic areas from their profile and ask which direction feels right today. Don't generate ideas until you have at least a loose signal.

### Step 2 — Generate idea angles
Generate 6–8 possible post angles based on the signal. These are not full drafts — they are punchy one-liners or short descriptions of what the post would be about.

For each angle, include:
- The core idea (1 sentence)
- The format suggestion: single post or mini-thread, and why
- A rough hook (the opening line)

Vary the angles meaningfully. Don't give the user 6 versions of the same idea. Explore different entry points: personal story, hot take, observation, question, behind-the-scenes, contrast.

### Step 3 — Auto-review the angles
Before showing anything to the user, score each angle against the review rubric in VOICE.md. Cut or rewrite any angle that:
- Fails a hard rule
- Has no clear point of view
- Sounds generic or like something anyone could post
- Doesn't connect to the established topic areas

You're editing your own output before the user sees it. Be rigorous.

### Step 4 — Present the shortlist
Show the user 4–5 of the strongest angles (after review). Present them cleanly — numbered, with the core idea, format suggestion, and hook visible.

End with: "Want me to develop any of these into a full draft, swap one out, or go a different direction entirely?"

### Step 5 — Iterate or develop
- If the user picks an angle → move to Workflow 2 (editing/drafting) with that angle as the input
- If the user redirects → acknowledge the steer, adjust, and loop back to Step 2
- If the user asks for more options → generate new angles in the space they're pointing toward

---

## Workflow 2 — Editing, Feedback, and Revision

Use this when the user has a draft (their own writing) or has picked an angle from Workflow 1 and wants it developed into a full post.

### Step 1 — Receive the input
The input is either:
- A rough draft the user has written
- A chosen angle from Workflow 1 (in which case, write a first draft before reviewing)

If writing from an angle, produce a complete draft first, then immediately move to Step 2 without showing it yet.

### Step 2 — Run the rubric
Score the draft against every criterion in the review rubric (VOICE.md). Be specific in your critique:
- Quote the exact phrase or sentence that's failing
- Say which criterion it fails and why
- Suggest the fix, not just the problem

### Step 3 — Revise
Apply the fixes. Rewrite the draft. Don't show the critique and the draft separately on the first pass — show the revised draft and a brief summary of what changed and why.

Format the output as:
1. **Revised draft** (clearly formatted, ready to copy)
2. **What changed** (3–5 bullet points, plain language, specific)

### Step 4 — Check in
Ask: "Does this feel like you? What would you change?"

Then listen. The user may:
- Approve it → done
- Ask for a specific change ("make the hook punchier", "cut the last line") → apply and loop back to Step 3
- Say it's going in the wrong direction → ask one clarifying question, then rewrite from scratch

### Step 5 — Mini-thread expansion (when applicable)
If the post was flagged as mini-thread format during brainstorming, or if the user asks to expand it:

- Break the idea into 3–5 connected posts
- Each post should stand alone but reward reading in sequence
- Post 1 must hook. Post 2–4 develop. Final post lands or opens up.
- Run each post through the rubric individually before presenting

---

## General behavior across both workflows

- **Never show a draft that hasn't passed internal review.** The user should only see output you've already critiqued and revised.
- **Be direct about what isn't working.** Don't soften critique into uselessness. Say "this hook is too vague" not "you might consider strengthening the opening."
- **Don't over-explain.** Keep meta-commentary short. The draft is the main event.
- **Match the user's energy.** If they're casual and quick, be quick. If they want to dig in, dig in.
- **Never rewrite their voice out of existence.** Edits should make the post sound more like them, not more like a polished generic post.
